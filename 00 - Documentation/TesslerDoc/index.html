Configuration

Tessler supports configuration through either the app.config file or using the fluent api.

App.config

Most configuration options can be set using the app.config, with the exception of the adapters. When configuration Tessler using the app.config, you need to add the following config section:

App.config

So now we can add the actual configuration elements.

Ajax wait interval
How long to wait when an ajax request is pending before polling for pending requests again. Specified in seconds.
Ajax wait time
How long to keep polling for pending ajax requests before timing out. If a request is still running after this, the test continues. Specified in seconds.
Browser
Which browser to use. Supported values are Chrome, Firefox and InternetExplorer.
Date format
Used for parsing dates when asserting them. Also used to format a DateTime object when typing it in a textfield.
Find element timeout
How long the driver will look for an element before timing out. This compensates for any page loading delays. Specified in seconds.
jQuery url
An url to a jQuery release, only used when jQuery is not already loaded. jQuery is internally used by Tessler.
Maximize browser
Whether to maximize the browser when running tests. Can either be true or false.
Not visible wait time
How long the driver will wait for an element to not be visible. This compensates for elements becoming invisible after a short delay. Specified in seconds.
Recycle browser
Whether to re-use browser instances between tests. When false, a new instance will be opened for each test.

Recycling the browser will save some time, but it also might cause some issues. First of all, cookies might prove to interfere with other tests. At the end of a test, cookies for the current domain will be removed, but in the case of identity providers some cookies might survive when they're on another domain. Additionally, alerts may mess up subsequent tests. Therefore, we recommend to only enable this on development environments.

Screenshots path
Where to save screenshots. This is a relative path starting at the test runner executable.
Strip namespace
Which part of the namespace to strip when creating a folder structure.

When saving screenshots, the framework will create a directory structure based on the namespace, class and test method name of the current test. Sometimes, that leads to a very deep directory structure, because of any recurring stuff added to the namespace.

If, for example, the fully qualified name of one of your tests is 'InfoSupport.C2Wmo.AutomatedTesting.AutoUITests.Portaal.Ambtenaar.Inloggen.InloggenSuccesvolTest', you can choose to strip the first bit off by setting this property to 'InfoSupport.C2Wmo.AutomatedTesting.AutoUITests'. That way, directory creation will start at 'Portaal' for this specific test.

Take screenshot
When to take screenshots of a test run. Supported values are Always, OnFail and Never.

The screenshot system has three modes: always, on fail and never. Always and never are self-explanatory. On fail always makes screenshots, but removes them when the test completes successfully. Every time a test is run, any residual screenshots from previous test runs will be removed.

Website url
Where to navigate the browser to after starting a test. Essentially the first page of the target application.

Fluent API

Tessler can also be configured using code only. Using the fluent API from TesslerState, all properties from the app.config can be set. Also, custom adapters can be configured with the code only approach.
